This guide walks you from a bare Sidekiq setup to a running uniqueness-protected job. Each step builds on the last — by the end you will have verified that duplicate jobs are silently dropped.

<Note>
  These steps assume you already have Sidekiq and Redis installed and running. If not, set those up before continuing.
</Note>

<Steps>
  <Step title="Add the gem to your Gemfile">
    Add sidekiq-unique-jobs to your `Gemfile` and run `bundle install`.

    ```ruby Gemfile
    gem "sidekiq-unique-jobs", "~> 9.0"
    ```

    ```bash
    bundle install
    ```
  </Step>

  <Step title="Configure the middleware">
    Create `config/initializers/sidekiq.rb` (or add to it if it already exists). Both the client and server middleware must be registered explicitly — the gem does not auto-load them.

    ```ruby config/initializers/sidekiq.rb
    Sidekiq.configure_client do |config|
      config.client_middleware do |chain|
        chain.add SidekiqUniqueJobs::Middleware::Client
      end
    end

    Sidekiq.configure_server do |config|
      config.client_middleware do |chain|
        chain.add SidekiqUniqueJobs::Middleware::Client
      end

      config.server_middleware do |chain|
        chain.add SidekiqUniqueJobs::Middleware::Server
      end

      SidekiqUniqueJobs::Server.configure(config)
    end
    ```

    `SidekiqUniqueJobs::Server.configure` registers startup and shutdown hooks that handle lock migration, the orphan reaper, and metrics flushing.
  </Step>

  <Step title="Add a lock to your first job">
    Add `lock: :until_executed` to any Sidekiq worker via `sidekiq_options`. This tells the gem to hold a Redis lock from the moment the job is enqueued until it finishes executing.

    ```ruby app/jobs/my_job.rb
    class MyJob
      include Sidekiq::Job

      sidekiq_options lock: :until_executed

      def perform(user_id)
        # Only one job per user_id runs at a time.
        # A second enqueue with the same user_id is dropped
        # until the first job completes.
      end
    end
    ```

    The lock key is derived from the worker class, queue name, and all arguments. Two calls with different `user_id` values produce different locks and both proceed normally.
  </Step>

  <Step title="Verify duplicate dropping">
    Start a Rails console and enqueue the same job twice with the same argument. Only the first call enqueues the job; the second is silently discarded.

    ```ruby
    MyJob.perform_async(42)  # => "abc123def456"  (job ID returned)
    MyJob.perform_async(42)  # => nil             (duplicate dropped)

    MyJob.perform_async(99)  # => "789xyz..."     (different arg = different lock)
    ```

    The default conflict strategy is `:log`, which discards the duplicate and writes a log line. To change this behavior, set `on_conflict` in `sidekiq_options`:

    ```ruby
    sidekiq_options lock: :until_executed,
                    on_conflict: :raise   # raises instead of silently dropping
    ```
  </Step>

  <Step title="Disable uniqueness in tests">
    Uniqueness enforcement in your test suite will cause jobs enqueued in one test to interfere with jobs in another. Disable it globally in your test helper:

    ```ruby spec/rails_helper.rb
    SidekiqUniqueJobs.config.enabled = false
    ```

    <Warning>
      Do not test the gem's uniqueness behavior from within your application's test suite. The gem has its own test suite that covers lock correctness. Testing it again in your app tests adds friction without value.
    </Warning>

    If you need uniqueness enabled for a specific test, re-enable it for that block:

    ```ruby
    around do |example|
      SidekiqUniqueJobs.use_config(enabled: true) { example.run }
    end
    ```
  </Step>
</Steps>
